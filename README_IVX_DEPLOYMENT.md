# IVX Deployment Readiness

This repository contains the Expo app and the IVX backend required for independent deployment through Render or AWS.

## Runtime entrypoints

- Backend server: `server.ts`
- Hono app: `backend/hono.ts`
- Health endpoint: `GET /health`
- Readiness endpoint: `GET /readiness`
- Owner AI endpoint: `POST /api/ivx/owner-ai`
- Supabase owner write endpoint: `POST /api/ivx/supabase/owner-action`
- Supabase inspection endpoints:
  - `GET /api/ivx/supabase/tables`
  - `GET /api/ivx/supabase/schema`
  - `GET /api/ivx/supabase/columns`
  - `GET /api/ivx/supabase/rls`
- IVX AI Brain tool endpoints:
  - `GET /api/ivx/ai-brain/tools`
  - `POST /api/ivx/ai-brain/tools`
  - `POST /api/ivx/ai-brain/tools/execute`
- Owner chat tool routing:
  - `POST /api/ivx/owner-ai` routes owner/developer status prompts to the AI Brain executor before generic chat.
- Owner-control readiness tooling:
  - `project_registry`
  - `project_surface_health`
  - `code_repo_control_status`
  - `deployment_readiness_matrix`
  - `owner_control_audit`
  - `owner_control_readiness_report`
- Route53 tooling endpoints:
  - `POST /api/aws/route53/audit`
  - `POST /api/aws/route53/upsert`

## Render deployment

The correct Render deployment is split into two public services:

1. Backend API Docker Web Service: `ivx-holdings-platform`, public URL expected at `https://api.ivxholding.com`.
2. Chat/frontend Static Site: `ivx-holdings-chat-frontend`, public URL expected at `https://chat.ivxholding.com`.

The root `render.yaml` defines both services. The backend uses `runtime: docker`, builds `./Dockerfile` from the repository root, starts with `node ./node_modules/tsx/dist/cli.mjs server.ts`, and checks `GET /health`. The frontend uses `runtime: static`, `rootDir: expo`, builds the Expo web export, and publishes `./dist`.

Exact backend Render settings:

| Setting | Value |
| --- | --- |
| Service name | `ivx-holdings-platform` |
| Service type | `Web Service` |
| Runtime / Language | `Docker` |
| Branch | `main` |
| Root directory | blank / repository root |
| Dockerfile path | `./Dockerfile` |
| Docker build context | `.` |
| Build Command | leave blank; Dockerfile performs the build |
| Start Command / Docker Command | `node ./node_modules/tsx/dist/cli.mjs server.ts` |
| Health Check Path | `/health` |
| Custom domain | `api.ivxholding.com` |
| Render subdomain policy | `disabled` after custom domain is attached/verified; use `enabled` temporarily if you need to test `*.onrender.com` directly or through the optional Worker origin bridge |
| Auto Deploy | On Commit / `autoDeployTrigger: commit` |
| Render private service env | `MINIO_HOST` from `fromService.name=minio`, `type=pserv`, `property=host`; `MINIO_PASSWORD` from `fromService.name=minio`, `type=pserv`, `envVarKey=MINIO_ROOT_PASSWORD` |
| Render Postgres env | `DATABASE_URL` from `fromDatabase.name=mydatabase`, `property=connectionString` |
| Render generated/manual secrets | `APP_SECRET` uses `generateValue: true`; `STRIPE_API_KEY` uses `sync: false` |
| Render env group | `fromGroup: my-env-group` is linked to the backend service if the group exists in Render |

Exact frontend Render settings:

| Setting | Value |
| --- | --- |
| Service name | `ivx-holdings-chat-frontend` |
| Service type | `Static Site` |
| Runtime | `Static` |
| Branch | `main` |
| Root directory | `expo` |
| Build command | `bun install --frozen-lockfile && bunx expo export --platform web` |
| Publish directory | `./dist` |
| Rewrite rule | `/*` -> `/index.html` |
| Custom domain | `chat.ivxholding.com` |
| Render subdomain policy | `disabled` after custom domain is attached/verified; use `enabled` temporarily if you need to test `*.onrender.com` directly or through the optional Worker origin bridge |
| Auto Deploy | On Commit / `autoDeployTrigger: commit` |

Manual Render steps:

1. Connect the GitHub repository to Render.
2. Create or sync a Blueprint from the repository root using `render.yaml`.
3. Confirm the backend service `ivx-holdings-platform` has the exact Docker settings above.
4. Confirm the frontend service `ivx-holdings-chat-frontend` has the exact Static Site settings above.
5. Confirm the backend service `ivx-holdings-platform` receives `DATABASE_URL` from `mydatabase`, `MINIO_HOST` from the private `minio` service host, and `MINIO_PASSWORD` from `minio` env var `MINIO_ROOT_PASSWORD` via the Blueprint references.
6. Confirm `APP_SECRET` is generated by Render, set `STRIPE_API_KEY` manually in Render, and link/create the Render Environment Group `my-env-group` if you want the `fromGroup` import active.
7. Set backend-only secrets on `ivx-holdings-platform` only; do not put backend secrets on the frontend static site.
8. Set public `EXPO_PUBLIC_*` variables on `ivx-holdings-chat-frontend` as needed for the web bundle.
9. In Render custom domains:
   - Add `api.ivxholding.com` only to `ivx-holdings-platform`.
   - Add `chat.ivxholding.com` only to `ivx-holdings-chat-frontend`.
10. After custom domains are attached, keep `renderSubdomainPolicy: disabled` if production should be reachable only through custom domains. If direct `*.onrender.com` testing or the optional Cloudflare Worker origin bridge is needed, temporarily set it to `enabled`.
11. Click Manual Deploy / Deploy latest commit for both services after env/domain changes.
12. Wait for Render to verify domains and issue TLS certificates.
13. Verify:
   - `curl -i https://api.ivxholding.com/health`
   - `curl -i https://chat.ivxholding.com/`
   - Expected: backend health returns HTTP 200 JSON `{ "ok": true, "status": "healthy" }`; frontend returns HTTP 200 HTML.

If Render provides targets like `ivx-holdings-platform.onrender.com` and `ivx-holdings-chat-frontend.onrender.com`, DNS should point each custom domain to its matching target unless Render displays a different target.

Recommended DNS records when using Render custom domains:

| Type | Name | Value/Target | Proxy |
| --- | --- | --- | --- |
| CNAME | api | Render-provided backend hostname, usually `ivx-holdings-platform.onrender.com` | DNS only if Cloudflare is used |
| CNAME | chat | Render-provided frontend hostname, usually `ivx-holdings-chat-frontend.onrender.com` | DNS only if Cloudflare is used |

## Optional Cloudflare Worker TLS/origin bridge

If Cloudflare is managing `ivxholding.com`, deploy `deploy/cloudflare/ivx-render-origin-worker.js` and attach routes:

- `api.ivxholding.com/*` -> Worker
- `chat.ivxholding.com/*` -> Worker

The Worker forwards `api.ivxholding.com` to `https://ivx-holdings-platform.onrender.com` and `chat.ivxholding.com` to `https://ivx-holdings-chat-frontend.onrender.com`, preserving the original host in `x-forwarded-host`. It stores no secrets.

Keep `IVX_WORKER_USE_RESOLVE_OVERRIDE` unset for the default direct-origin proxy mode. Only set `IVX_WORKER_USE_RESOLVE_OVERRIDE=true` if the Cloudflare zone/DNS setup specifically requires preserving the incoming URL host while overriding DNS resolution.

This Worker can repair Cloudflare-side TLS/custom-domain routing, but it cannot create or activate a missing Render service. If `renderSubdomainPolicy` is `disabled`, Render intentionally returns 404 for default `*.onrender.com` hostnames; in that mode, production proof must use `api.ivxholding.com` and `chat.ivxholding.com` directly, or the subdomain policy must be temporarily re-enabled for Worker/default-origin testing.

## AWS deployment option

The backend can also run on AWS using a container service such as ECS/Fargate or EC2 with Docker. The AI Brain executor includes read-only AWS diagnostics for STS identity, IAM, S3, CloudFront, Route53, ACM certificates, EC2, ECS, ELB/ALB target groups, SSM parameter metadata, AWS Organizations, and an aggregate deployment inventory.

Minimum AWS path:

1. Build the Docker image from the repository root.
2. Run the container with `PORT=3000` and `HOST=0.0.0.0`.
3. Attach the service to an HTTPS load balancer or reverse proxy.
4. Point `api.ivxholding.com` to the active load balancer target.
5. Point `chat.ivxholding.com` to either the same backend web export or a separate static web host.
6. Configure TLS with ACM or the platform certificate manager.
7. Verify `/health` before testing Owner AI.

## Verification commands

From repository root:

```bash
bun install
bunx tsc --noEmit --pretty false
bun server.ts
```

Render itself should not use a Node/Bun native runtime for this service. Render should use Docker, leave the Build Command blank, and run the Docker Command:

```bash
node ./node_modules/tsx/dist/cli.mjs server.ts
```

From another terminal:

```bash
curl -i http://localhost:3000/health
curl -i http://localhost:3000/readiness
curl -i https://api.ivxholding.com/health
```

Owner-only routes require a valid IVX/Supabase owner session token in the `Authorization` header.

## Production readiness checklist

- GitHub repository contains root backend files: `server.ts`, `backend/`, `Dockerfile`, `render.yaml`, `tsconfig.json`, `bun.lock`, `IVX_OWNER_CONTROL_READINESS.md`.
- Render/AWS runtime has the required environment variables.
- Supabase SQL has been applied.
- `https://api.ivxholding.com/health` returns HTTP 200.
- `POST /api/ivx/owner-ai` is reachable after health is green and routes developer/status prompts to AI Brain tools.
- DNS resolves to the active deployment target.
- TLS certificate covers `api.ivxholding.com` and `chat.ivxholding.com`.
